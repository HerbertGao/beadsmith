# tests/golden/

Committed byte golden masters that freeze the Phase-1 engine's output (milestone
M7). They turn determinism (CLAUDE.md rule 2) from a documented promise into an
executable CI gate: an intentional algorithm change — or a dependency bump that
changes output bytes — fails loudly here instead of drifting silently.

The golden test lives at `crates/bead-cli/tests/golden.rs`. It drives the library
API (`generate_pattern`) and compares the four products against the masters in
this directory. Because the CLI writes those same byte blobs verbatim via
`fs::write`, freezing the library output transitively covers the CLI contract
(rule 5).

## Fixed setup

The golden masters are produced from a single fixed input + fixed settings.
Changing any of these changes the masters — the settings are part of the golden.

- **Fixture:** `samples/gradient.png` — a committed 32x40 image.
- **Resize:** to **16x20**, filter **Lanczos3** (the real engine default — the
  path users actually get; not a stand-in like `Nearest`).
- **Cell size:** `cell_size 10`.
- **Palette:** `palettes/artkal_s.json` (`artkal_s`).

The fixture is the committed 32x40 `gradient.png` scaled **to** 16x20. Because
`16:20 == 32:40`, the crop is a no-op — do not regenerate a separate 16x20
fixture.

These settings are pinned in the test code, not exposed as CLI flags.

## Golden files

Four masters, one per product the CLI writes. They are generated by the golden
test's `BLESS` regeneration path (see `crates/bead-cli/tests/golden.rs`):

- `pattern.json`
- `summary.txt`
- `preview.png`
- `grid.png`

## Canonical platform: arm64 Linux

The byte masters are valid **only on the canonical platform (arm64 Linux; CI
reference runner = `ubuntu-24.04-arm`)**. The `Lanczos3` resize path runs
floating-point math whose cross-platform / cross-architecture bit-for-bit result
is not guaranteed (the weight kernel goes through `f32::sin`, which is not
specified to be identical across architectures / libm implementations). arm64 is
chosen because the engine's production target is mobile arm64 (iOS / Android),
and Apple Silicon dev machines can bless via a *native* arm64 container (no
emulation). So:

- **arm64 Linux:** the four products are asserted **byte-for-byte** against these
  masters.
- **x86-64 Linux / macOS / Windows:** the same test runs but asserts only
  float-independent **structural invariants** (PNG decodes, `preview` is 160x200,
  `pattern.json` key order / counts / indices, `summary.txt` structure,
  `grid.png` constant pixels). It does **not** compare bytes.

Therefore the masters **must be generated on the canonical (arm64 Linux)
platform**. Do not bless on macOS, Windows, or x86-64 Linux — that would commit
non-canonical bytes and turn the canonical (`ubuntu-24.04-arm`) CI assertion red.

Note: golden is a *regression tripwire*, not a production-fidelity check — it
cannot match production bytes, because iOS (Apple libm) and Android (Bionic libm)
differ even though both are arm64. The canonical platform is chosen only for
stable, reproducible, easy-to-bless masters.

## Regenerating the golden masters

When you intentionally change something that alters the output bytes, regenerate
the masters **on the canonical (arm64 Linux) platform**. On an Apple Silicon Mac
this runs in a *native* arm64 container (no emulation):

```bash
docker run --rm --platform linux/arm64 -v "$PWD":/work -w /work \
  -e CARGO_TARGET_DIR=/tmp/ct -e CARGO_HOME=/tmp/ch -e RUSTUP_HOME=/tmp/rh \
  ubuntu:24.04 bash -lc '
    apt-get update -qq && apt-get install -y -qq curl ca-certificates build-essential pkg-config
    curl --proto "=https" --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y --profile minimal --default-toolchain stable
    . /tmp/ch/env
    BLESS=1 cargo test --locked -p bead-cli --test golden'
```

Or directly on an arm64 Linux box / a `ubuntu-24.04-arm` CI runner:

```bash
BLESS=1 cargo test -p bead-cli --test golden
```

This rewrites the four files above. Then review `git diff tests/golden/` so the
byte change is a deliberate, audited action. Re-run without `BLESS` to confirm
the read-only comparison passes:

```bash
cargo test -p bead-cli --test golden
```

## golden + `--locked` = dependency-bump guard

`Cargo.lock` is committed and CI runs `cargo test --locked`, so dependency
upgrades are an auditable diff. Golden raises that to a behavior-level tripwire:
if an `image` / `png` bump changes resize output or PNG encoding bytes, the
golden test goes red on the canonical platform in the bump PR — loud, deliberate,
blocking, instead of a silent drift. A bump that changes bytes must be
consciously blessed.

## `.actual` side-cars

On a mismatch the test writes the actual output next to the master as
`<name>.actual` (e.g. `pattern.json.actual`) for inspection — diff `<name>`
against `<name>.actual`. These side-cars are already excluded from the repo via
`.gitignore` (`tests/golden/*.actual`); they are never committed.
