## 决策

### D1：内置色卡用 `include_str!` 编译期嵌入，而非运行时扫目录
`palette list` 需要一份「有哪些内置色卡」的清单。选择在 CLI 层用
`include_str!("../../../palettes/<name>.json")` 把 14 份 JSON 编译期嵌入二进制。

- **替代方案：运行时扫 `palettes/` 目录**。**否决理由**：装好的二进制不知道 repo 在哪，
  依赖 CWD 或写死路径都脆；且会给 CLI 引入「约定俗成的目录位置」这一隐性 fs 契约。内置则
  装到任何位置都可列、结果确定。
- **替代方案：让 `bead-core` 暴露内置色卡表**。**否决理由**：违反规则 1（`bead-core` 无
  fs / 无内置资源假设）；色卡资产归属 CLI/App 打包层，不属引擎。

### D2：品牌名与色数由 `load_palette` 解析，不在表里硬编码
表只存 `(id, 内置 JSON)`；`brand` 与色数遍历时经 `load_palette` 得出。

- **替代方案：表里直接写死 `(id, brand, count)`**（如 App 注册表存 `brand`）。**否决理由**：
  App 存 `brand` 是为「色卡行/弹窗首帧同步显示」且有测试锁定与 JSON 一致；CLI 无此约束，
  纯批处理输出无需预存，解析更省更稳、也不会与 JSON 漂移。

## 确定性

纯新增只读列举命令，不经 `pipeline`、不产出任何 pattern/统计/渲染、不动 golden。同一内置
色卡集恒产出同一清单——逐字节可复现。
